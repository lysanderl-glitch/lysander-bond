> 版本：v1.1 | 编制日期：2026-05-04 | 编制团队：OBS · Harness Ops · Product Ops · Content Ops · AI/ML · 智囊团

## 核心结论（Executive Summary）

**我们正在面对的真实问题**：今天每个人都在用 AI，但每次都从零开始——AI 不记得上次说过什么，不知道公司有哪些规则，也无法保证输出质量稳定。业务专家花了大量时间"调教 AI"，而不是用 AI 解决业务问题。更深层的问题是：AI 的能力、判断和知识，没有办法在组织里沉淀下来。

**Synapse 解决什么问题**：让业务专家（非工程师）用业务语言驱动 AI 完成复杂专业任务，过程可审计、结果可积累、能力可复制。不需要懂 Prompt Engineering，不需要每次从头解释背景，不需要手动把关每一步细节。

**带来什么价值**：Synapse Owner 参与度从 100% 降至 0%（仅接收 Slack 结果通知）；Phoenix PMO 自动化产品从产品规划（D1-D8 全部 AI 自主完成）到部署上线，Synapse Owner 只在三个节点介入（方案审批、架构审批、Stage 2 暂停处理），其余全程由 Lysander 与委员会自主执行；超过 50 个 D-编号决策完整归档，团队知识真正可积累、可检索、可传承。

## 一、体系架构与核心能力

Synapse 体系由五层组成，形成"知识 ←突触→ 决策 ←突触→ 执行"的完整信息流转网络：

```
├── 知识层 (OBS 第二大脑)   — 团队记忆的外部化
├── 控制层 (Harness)        — AI 行为的工程化约束
├── 执行层 (Multi-Agents)   — 专业分工的自动化执行
├── 决策层 (四级制)         — 权责清晰的分级授权
└── 进化层 (情报闭环管线)  — 系统自我学习与迭代
```

### 1.1 Harness Engineering：让 AI 可管理、可约束、可进化

**核心价值**：像工程师管理代码一样管理 AI 行为。

Harness Engineering 是 2026 年由 Martin Fowler、Red Hat 等权威机构确认的 AI 工程新范式，核心公式为：

> **Agent = Model（模型） + Harness（控制层）**

即：竞争优势不在于谁的模型最大，而在于谁的 Harness 最有效。Synapse 将这一理念完整工程化落地，构建了三层控制机制：

**Guides（前馈控制）**——在 AI 行动之前，预先定义边界：
- `CLAUDE.md`：主配置文件，定义角色、执行链、约束体系（当前 322 行，上限 350 行）
- `organization.yaml`：团队架构与路由规则，75 条关键词自动映射到对应专家
- `.claude/harness/` 模块库：18 个专项规则文件（凭证管理、派单模板、HR 管理等）

**Workflow（结构化流程）**——执行链 v2.0，8 步强制流程：
```
【开场】身份确认 → 【0.5】Synapse 承接（6步）→ 【①】分级（S/M/L）
→ 【②】路由派单 → 【③】QA 质量门禁（≥85/100）→ 【③.5】PQG 过滤
→ 【④】结果交付
```

**Constraints（约束系统）**——CEO 执行禁区 P0 硬性约束：
- 主对话禁止直接调用 Bash/Edit/Write（防止 AI 越权执行）
- 任何任务必须经过 Synapse 承接和派单（可审计的唯一合法执行路径）
- 审计日志 `logs/ceo-guard-audit.log` 自动记录所有工具调用

**实际效果**：通过 PreToolUse hook 技术防护，系统能在代码层面拦截任何未授权的 AI 直接执行行为，使 AI 团队的所有操作都处于可审计、可追溯的管控范围内。

### 1.2 知识管理体系（OBS 第二大脑）

**核心价值**：让团队记忆从"人脑"迁移到"系统"，实现知识的积累、检索与传承。

**Obsidian 知识图谱**——结构化知识库，覆盖 7 个领域：

| 知识域 | 文件数 | 内容 |
|--------|--------|------|
| 产品知识 | 70+ | 产品线档案、PRD、委员会记录 |
| 流程知识 | 27+ | SOP、最佳实践、运行手册 |
| 决策知识 | 50+ | D-编号决策归档、战略分析 |
| 团队知识 | 69 | Agent 人员卡片、HR 记录 |
| 内容管线 | 持续增长 | 博客草稿、已发布文章 |
| 情报报告 | 每日生成 | 每日技术动态、行动报告 |
| 分析报告 | 按需生成 | QA 报告、Sprint 报告 |

**跨会话状态持久化**——解决"AI 每次都要重新解释"的根本问题：
- `agent-CEO/config/active_tasks.yaml`：存储当前进行中任务的 stage/blocker/next_step
- 每次新会话开始自动读取，任务从断点续接，无需重新交代背景
- 授权颗粒度是目标（Objective），跨会话持有，不因切换重启

**Memory 系统**——跨会话经验积累（35 条已沉淀）：
- `feedback_*.md`：操作偏好与工作方式反馈（21 条）
- `project_*.md`：产品线状态与技术配置记录（14 条）
- 每次发现新的工作模式偏好，自动更新，下次会话即生效

**决策归档强制机制**（P0，2026-04-27 Synapse Owner 批准）：
- 所有 L3+ 决策必须以 D-编号文档归档至 `obs/04-decision-knowledge/decision-log/`
- CI 脚本 `scripts/check_decision_log.py` 自动拦截无 D-编号的决策性提交
- 截至 2026-05-03 已归档 50+ 条 D-编号决策，形成完整的组织决策历史

**D8 产品规划阶段**（8 步严格流程，4 天内完成，全程 AI 自主执行）：

**Stage Gate 管理**（分 3 阶段验收）：

**关键结果**：Synapse Owner 参与度从"全程参与"降至"仅看 Slack 结果通知"，部署时间从 30+ 分钟人工操作降至 5 分钟全自动。7 项 D-编号决策全部入库，端到端实证 artifact 完整（Monday board、Notion charter pages、团队邀请全部真实生效）。

### 1.5 自我进化能力

**核心价值**：系统不依赖人工干预，持续优化自身的规则、流程和知识储备。

**周维度 Harness 审查**（P0，2026-04-27 Synapse Owner 批准）：
- 执行时间：每周日 23:00 Dubai 时间
- 审查维度（6 个）：CLAUDE.md 行数 / Agent 卡评分 / frontmatter 合规 / fact-SSOT 漂移 / 陈旧任务 / 决策与执行链
- 执行者：harness_engineer + execution_auditor 独立执行（非 Synapse 自评，避免自评者陷阱）
- 输出：周报入 `obs/01-team-knowledge/HR/weekly-audit/`
- 告警机制：连续 2 周缺生成 → L4 上报 Synapse Owner

**PQG 自改进机制**（2026-04-30 启用）：
- 基于对 7 个主要会话的 141 个真实中断事件的挖掘分析，识别 6 类高频中断场景
- 每周通过新的会话数据迭代拦截规则
- 从分析激活至今：系统性消除约 40/70 类不必要中断
- 实测评分：整体 8.4/10（integration_qa 圆桌审查）

**规则时间戳 + 熵增预算**——防止规则无限膨胀：
- 每个新规则标注 `# [ADDED: YYYY-MM-DD]`
- 超过 180 天未被重申的规则自动进入废弃候选
- CLAUDE.md 行数上限 350 行（已固化约束，超限须先删减再添加）
- Phase 2 完成后收紧至 300 行

**情报闭环管线**——持续感知外部环境变化：
- 每日 8:00 Dubai：情报日报生成（AI 生成技术动态摘要）
- 每日 10:00 Dubai：情报行动执行（扫描 OBJ 关键词 → 评估相关性 → 执行相关任务 → Slack 汇报）
- 信息流：外部信号 → 知识沉淀 → 决策支持 → 执行迭代

### 1.6 Token 优化策略（调研完成，方案待批）

**背景**：中文 token 消耗约为英文的 2.0 倍（实测范围 1.7-2.4x），System Prompt 每次对话固定消耗约 17,664 tokens。

**调研结论**——方案 C（5/5 推荐分）：
- 将 Harness 文档（CLAUDE.md + 13 个 harness 文件）全部英文化
- Agent-to-Agent 派单通信增加英文约束
- active_tasks.yaml 状态字段改为英文存储
- Synapse Owner 与 Synapse 界面保持中文，零感知迁移
- **预计节省：每次对话 40-55% token 消耗**

## 三、推广价值与应用场景

从实际运营结果来看，Synapse 体系已在以下维度产生可量化的价值：

**执行质量保障**：

- 所有代码变更必须经过 integration_qa 验证
- 重大架构变更需圆桌 9/10 多数票通过

**跨会话连续性**：
- 通过 `active_tasks.yaml` 实现任务断点续接
- 35 条 Memory 记录系统性保存工作模式和项目状态
- 新会话启动时，系统自动汇报进行中任务并继续（无需重新交代背景）

**实测成果列举**：
- Phoenix 产品交付：跨 4 天多会话持续执行，Synapse Owner 仅在 3 个关键节点参与
- 管线恢复：5 天内容缺口 + 12 篇博客文章，单日全部恢复上线
- Synapse Owner 打断次数：从历史 141 次/7 会话（约 20 次/会话）降至接近 0

### 3.2 推广场景一：项目管理领域（Spike 案例）

**业务背景**：Spike 负责项目管理相关培训，需要持续基于真实系统变更更新培训材料——这是一个高频、高重复度的内容生产任务，目前完全依赖人工。

**核心痛点**：
- 系统每次更新，培训材料就需要重写（工作量与系统迭代速度正相关）
- 培训内容需要以业务视角（而非技术视角）描述，但技术团队无法保证语言风格
- 审核成本高：Spike 需要从零审核，而非基于可靠基础上做增量修改

**Synapse 解法（MECE 分析）**：

| 维度 | 问题 | Synapse 能力 | 交付成果 |
|------|------|-------------|---------|
| 知识沉淀 | 系统变更记录分散 | OBS 知识库自动跟踪 | 变更日志 → 结构化知识条目 |
| 内容生成 | 培训材料须以业务视角写作 | Content Ops 团队（content_creator + content_training） | 业务视角培训文档初稿 |
| 质量保证 | 培训内容准确性需专家审核 | QA 门禁 ≥85 + 圆桌审查 | Spike 只审核最终产出，不参与 AI 交互 |
| 版本管理 | 旧材料 vs 新材料版本混淆 | D-编号归档 + CI 版本锁定 | 每次培训材料更新自动版本化 |

**Spike 的新工作模式**：
1. 描述业务需求（"这次更新了 X 功能，需要更新 PMO 入门培训第 3 章"）
2. Synapse 体系自动生成符合业务逻辑的培训材料初稿
3. Spike 审核最终成果并提出修改意见
4. 循环迭代直至验收，**全程不参与 AI 交互过程**

**类比 Phoenix 模式**：Synapse Owner 只参与 D8 审批包（目标与最终验收），Spike 也只参与业务需求输入和最终审核。

### 3.3 推广场景二：QA 测试标准化（Suzy 案例）

**业务背景**：Suzy 负责 QA 测试，需维护回归测试脚本和测试相关培训材料——两项工作都依赖对系统逻辑的深度理解，且都需要与代码变更保持同步。

**核心痛点**：
- 测试用例与业务场景脱节（测试团队知道"怎么测"，不知道"为什么测"）
- 培训材料与实际测试流程不一致（知识分散在不同人的脑子里）
- 新人上手成本高（无标准化的测试知识库）

**Synapse 解法（MECE 分析）**：

| 维度 | 问题 | Synapse 能力 | 交付成果 |
|------|------|-------------|---------|
| 知识体系 | 测试用例与业务场景无绑定 | OBS 双向绑定（测试 ↔ 业务场景） | 可检索的测试 ↔ 业务映射矩阵 |
| 测试标准化 | 无统一的回归路径定义 | Stage Gate 模式复用（仿 Phoenix S1/S2/S3） | 标准化回归测试路径（3 个 Gate） |
| 培训内容 | 测试培训与实际流程脱节 | Content Ops 流水线直接复用 | Suzy 审核级别的培训初稿 |
| 质量门禁 | 测试报告质量参差不齐 | integration_qa 自动 QA（≥85/100） | 每次输出有量化质量评分 |

**关键差异化**：Synapse 的 Stage Gate 模式（在 Phoenix 中已验证）可以直接复用到测试标准化场景。测试回归流程分 3 阶段（Stage 1：用例建库 / Stage 2：自动化脚本 / Stage 3：培训材料），每个 Stage 有清晰的验收门禁。

**Suzy 的新工作模式**：
1. 定义业务验收标准（"这个功能的核心业务逻辑是 X，边界条件是 Y"）
2. Synapse 生成测试用例矩阵 + 测试脚本初稿
3. Suzy 执行验收：只需验证"测试是否覆盖了我关心的业务场景"
4. Stage Gate 通过后，培训材料自动从测试用例生成

### 3.4 非软件工程人员的能力边界提升

**核心价值主张**：Synapse 让非技术人员获得"管理 AI 团队"的能力，而非仅仅"使用 AI 工具"。

**传统模式 vs Synapse 模式对比**：

| 维度 | 传统 AI 使用 | Synapse 体系 |
|------|-------------|-------------|
| 所需技能 | Prompt Engineering | 业务视角描述需求 |
| 交互方式 | 每次从零开始对话 | 系统记忆持续积累 |
| 质量控制 | 靠使用者主观判断 | ≥85/100 数字门禁 |
| 决策追溯 | 无记录 | D-编号完整归档 |
| 能力扩展 | 受限于个人 Prompt 能力 | 团队专业分工（69 名 Agent）|
| 错误纠正 | 每次对话重新纠正 | Memory 系统永久沉淀 |

**Harness 护栏的意义**：非技术人员不需要懂 Prompt Engineering——CLAUDE.md 和 `.claude/harness/` 的规则体系已经把"如何正确使用 AI"工程化为系统约束，使用者只需以业务视角驱动，系统保证质量和合规。

**实际能力边界提升**：
- 从"使用 AI 工具"（帮我写个文案）
- 到"管理 AI 团队"（我们的营销目标是 X，受众是 Y，Content Ops 团队按我们的品牌风格生成 + 审核 + 发布）
- 本质跃迁：从"个人效率工具"升级为"组织运营基础设施"
