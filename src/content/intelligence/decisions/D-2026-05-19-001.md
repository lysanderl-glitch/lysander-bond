# D-2026-05-19-001: WF-09 Slack 通知频道修复

## 问题描述

`intel-daily.yml` 和 `intel-action.yml` GHA 工作流中，"Notify success/failure via Slack" 步骤
持续返回 `{"ok":false,"error":"channel_not_found"}`，总裁从未收到情报管线推送通知。

总裁确认：所有 Slack 通知应走 n8n WF-09（`atit1zW3VYUL54CJ`），不绕过。

## 根因分析

**双重通知问题 + 载荷格式不匹配：**

GHA 工作流存在两层 Slack 通知：

| 层级 | 位置 | 载荷格式 | 问题 |
|------|------|----------|------|
| 层 1（Python 脚本） | `daily_agent.py` / `action_agent.py` 内 `notify_slack()` | `{recipient, title, body, source, priority}` | ✅ 正确，符合 WF-09 schema |
| 层 2（GHA step） | `intel-daily.yml` L115-L139 / `intel-action.yml` L123-L147 | `{text, run_url}` | ❌ 错误，WF-09 不认识 `text` 字段 |

**WF-09 Parse Recipient 节点**处理逻辑：收到 payload 后寻找 `recipient` 字段，若字段缺失则
路由失败，向 Slack API 传入空/无效 channel，导致 `channel_not_found`。

GHA step 中的 curl 调用直接 POST 到 WF-09 Webhook（`SLACK_WEBHOOK_N8N` 指向
`https://n8n.lysander.bond/webhook/notify`），但 payload 格式为原生 Slack Incoming Webhook
格式（`text` 字段），WF-09 无法正确解析。

**额外问题：** GHA step 发送的是第二次通知，与 Python 脚本内的 `notify_slack()` 重复，
若两者都成功会导致总裁收到双倍消息。

## 决策

**移除** `intel-daily.yml` 和 `intel-action.yml` 中所有直接 curl Slack 的 GHA step
（"Notify success via Slack" 和 "Notify failure via Slack"），原因：

1. Python 脚本（`daily_agent.py` / `action_agent.py`）已内含 `notify_slack()` 调用，载荷格式
   正确，业务级 `ok=true` 断言完整，日志可见 — 功能完全覆盖
2. GHA step 载荷格式与 WF-09 schema 不兼容，是 `channel_not_found` 的直接根因
3. 保留 GHA step 会产生重复通知（同一管线跑 2 次 notify）
